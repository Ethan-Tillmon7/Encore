Detailed screen-by-screen audit

Click any screen to expand its full UX analysis.

HomeView — Festival Hub
existsoverloaded
▶
What it does well
Single-screen festival context — crew, trip, schedule at a glance
Day picker for schedule is a strong pattern
Conflict banner surfaced inline is smart
Problems to fix
No single dominant action — the screen tries to be everything
Trip card taps to nothing — stubs without affordance hurt trust
No onboarding hook — new users see empty crew and schedule
Festival header is static — no countdown, no live context
Lineup entry is a single row — discovery deserves its own tab
Recommended changes
→
Add live countdown to festival header. "3 days, 14 hours until gates open" transforms a static label into a meaningful signal.
→
Replace "Browse Lineup" row with a "Next up" card. Show the user's next scheduled set prominently — the most relevant piece of info during the festival.
→
Make Trip card stubs visible as "coming soon" with disabled state. Or remove until implemented — dead taps undermine confidence.
→
Add empty states with CTAs. Empty crew: "Invite your squad →". Empty schedule: "Browse the lineup →". These guide new users toward the right action.
LineupView — Timetable Grid
existscore feature
▶
What it does well
2-axis scroll timetable is the right pattern for a festival lineup
Absolute positioning by time is more accurate than a list
Day picker + stage columns maps to real-world mental model
Problems to fix
No search or filter entry point in this view
No "now" indicator — critical during the festival
No way to see your scheduled sets highlighted in the grid
No toggle between grid and list view for accessibility
Crew overlap on blocks not visible in grid
Recommended changes
→
Add a "now" line. A horizontal rule at the current time that auto-scrolls into view on load. Table stakes for any live schedule app.
→
Highlight user's scheduled sets in the grid. A filled border or checkmark overlay on blocks that are in the user's schedule.
→
Show crew avatar stacks on set blocks. Tiny crew member initials on a block tell the user at a glance "your crew is watching this."
→
Add search bar + tier filter chips to LineupView toolbar. MatchTier pills and search exist in LineupStore but have no UI in LineupView.
→
Add a list view toggle. Grid is powerful but hard to scan alphabetically. List mode is where ArtistCardView finally gets used.
ArtistDetailView — Artist Sheet
existssolid foundation
▶
What it does well
Clear info hierarchy: tier → set info → crew → setlist → CTA
safeAreaInset bottom actions keep CTAs always visible
Crew attendance section is a strong social hook
Problems to fix
No artist image / visual hero — text-only top feels sparse
Setlist is 7 hardcoded songs — no source attribution
Sounds-like list has no interactivity — could link to similar artists
No pre-conflict warning before the user adds a set
Recommended changes
→
Add a visual hero area. Gradient bleed from tier color + artist name in large type. Tier-colored gradient + genre tags makes the top half feel designed, not empty.
→
Add "Similar artists in lineup" row. soundsLike values mapped to real artists in MockData. Tapping opens that artist's detail sheet — serendipitous discovery.
→
Make conflict state visible before adding. If this set conflicts with one already scheduled, show a warning inside the detail sheet before the user taps "Add."
FestivalMapView — Venue Map
existsburied
▶
What it does well
initialStage parameter for direct deep-linking from artist detail
Slide-up stage card with current/next act is a great pattern
Amenity toggles (water/medical/charging) are useful
Critical problems
Map is NOT in the tab bar — only reachable via lineup
Location sharing toggle does nothing (CoreLocation not wired)
Walk time is hardcoded "~8 min" regardless of position
No meetup pin UI — MeetupPin model exists but has no view
No crew member locations shown on map
Recommended changes
→
Map must be a top-level tab. During a festival, navigation is a primary use case. Burying the map 3 taps deep is a critical UX failure for real-world usage.
→
Add meetup pin UI. MeetupPin model and CrewStore.dropPin/removePin exist. The map needs a long-press gesture to drop a pin and a pin management sheet.
→
Design crew location layer now. Even as a stub, the map should show placeholder crew avatars so the UI is ready for Phase 2 CoreLocation data.
ConflictResolverView — Conflict Modal
existswell-scoped
▶
Side-by-side with Keep A / Keep B / Decide Later is clear and decisive
.medium detent is correct — doesn't need full screen
No crew context — crew attendance should influence the choice
"Decide Later" accumulates unresolved conflicts with no follow-up
Only resolves one conflict at a time — no batch review
Recommended changes
→
Add crew attendance to each side. Show crew member avatars under each set card. Knowing "Marcus + Zoe are seeing Set A" is the most useful signal for the decision.
→
Add a conflict queue indicator. If there are 3 conflicts, show "1 of 3" and allow stepping through all of them in one session.

Critical UX gaps

Issues that would cause real confusion or friction for an actual festival attendee.

P0 — Must fix before any real usage
🔴
Map is not tab-accessible. During a festival, attendees need the map constantly — not buried 3 taps into the Home tab. This is the single most impactful navigation fix.
🔴
No onboarding or first-launch state. A new user opens the app and sees empty crew, empty schedule, no Spotify match scores. There's no guidance on what to do first.
🔴
No "now" time indicator in timetable. Without a current-time line in the grid, users cannot quickly orient to what's happening right now. Essential for real festival use.
🔴
Dead taps everywhere. Trip card (Travel/Packing/Expenses), Crew Invite, Edit Profile, Notifications — 6+ interactive elements that do nothing. These erode trust immediately.
P1 — Fix before launch
🟡
Crew has no dedicated space. CrewStore has rich data (schedules, online status, last seen stage) but it's only surfaced in a small card on HomeView and a section in ArtistDetailView.
🟡
ArtistCardView is orphaned. Built for a DiscoverView that was removed. Needs a home — either inside LineupView as a list-mode toggle, or in a restored DiscoverView.
🟡
Scheduled sets have no visual feedback in the timetable. After adding a set, the block looks identical. A checkmark or filled border on scheduled blocks is critical feedback.
🟡
No pre-conflict warning. The conflict is only surfaced after adding a set. ArtistDetailView should check for conflicts before the user taps "Add" and warn them inline.
🟡
Schedule persistence is in-memory only. Every app launch resets the schedule. UserDefaults encoding of scheduledSets is a Phase 1 must.
P2 — Quality improvements
🔵
Crew overlap not visible in timetable blocks. The timetable grid should show crew member initials on set blocks where crew members are attending — a key feature differentiator.
🔵
No haptic feedback patterns defined. Adding a set, resolving a conflict, dropping a map pin — all warrant haptic feedback via UIImpactFeedbackGenerator.
🔵
HomeView festival header is static. Pre-festival: show a countdown. During festival: show "Day 2 of 4." Post-festival: show a recap prompt. Header should be context-aware.
🔵
No artist image strategy. A placeholder gradient strategy (derived from tier color + name hash) would make the app feel far more polished without requiring real artist images.

Views to build

New screens required to complete the app's feature surface, ordered by priority.

P0OnboardingView
Current phase
One job: Get the user set up with Spotify (optional), a display name, and their first crew — before they see the empty home screen.

  • Screen 1: Welcome — festival name + dates, "Let's get you ready" CTA
  • Screen 2: Connect Spotify (skip option) — explains match scoring, shows preview tier distribution
  • Screen 3: Your name — text field for display name used in crew
  • Screen 4: Find your crew — "Create crew" or "Join with code" options, or "Go solo" skip
  • Screen 5: Done — deep-link to lineup with "Start adding sets"

Only shown on first launch. Store completion flag in UserDefaults.
P0CrewView — Crew Hub
Current phase
One job: Full crew management and coordination in one place.

  • Header: Crew name + member count + invite code
  • Members section: Each member card — avatar, name, online indicator, last seen stage, their schedule (expandable)
  • Shared schedule: Timeline of all sets any crew member is attending, with member overlap indicators
  • Meetup pins: List of active map pins with creator + coordinates → tap to open map
  • Group chat: Phase 2, design stub now

Lives as its own tab. The Group Card in HomeView becomes a compressed preview that deep-links here.
P0CrewInviteView
Current phase
One job: Create a crew or join an existing one via invite code.

  • Mode A — Create: Text field for crew name → generates 6-char invite code → share sheet with code + QR (Phase 2)
  • Mode B — Join: 6-field code entry (one char per box) → validates → joins crew
  • Presented as a sheet from HomeView Group Card or from CrewView empty state
  • Shows an error state when code is invalid
P1DiscoverView — Artist Browse
Phase 1
One job: Browse and search the full lineup by tier, genre, or stage — without the timetable grid.

  • Toolbar: Search bar + tier filter chips (Must-see / Worth checking / Explore / All)
  • Sort modes: By tier (default), by name, by stage, by day
  • List: ArtistCardView rows — this is where the orphaned component finally lives
  • Empty state: "No artists match this filter" with reset button

Restores the removed DiscoverView with a cleaner filter model using LineupStore.filteredSets.
P1TripDetailViews — Travel, Packing, Expenses
Phase 1
Three sheets presented from the Trip Card on HomeView:

  • TravelView: Arrival/departure info, campsite location, car/transit details. Simple form + save to UserDefaults.
  • PackingView: Checklist-style packing list with Bonnaroo-specific preloaded defaults.
  • ExpensesView: Simple ledger — add line items with amount + description, running total, optional per-person split.
P1NotificationsView
Phase 1
One job: Configure which alerts the user wants to receive.

  • Set reminders: toggle + time picker (15 / 30 / 60 min before)
  • Conflict alerts: notify when a newly added set conflicts
  • Crew alerts: when a crew member changes their schedule
  • UNUserNotificationCenter permission request on first toggle
  • Presented as a sheet from ProfileView Notifications row
P1EditProfileView
Phase 1
One job: Edit display name and avatar color used in crew.

  • Name text field (used as CrewMember.name)
  • Color picker — grid of preset hex colors matching the avatar color system
  • Initials preview that updates live as name changes
  • Save → updates CrewStore and UserDefaults
P2GroupChatView
Phase 2
One job: Real-time text chat within a crew via Supabase Realtime.

  • Standard chat bubble layout — sent right, received left
  • System messages: "Marcus added Hozier to their schedule"
  • Deep-link previews: set block previews inside chat when a set is shared
  • Design stub can be built now; wire up in Phase 2

Navigation architecture

Current vs. recommended. The core change is moving from 2 tabs to 5 tabs, giving Map and Crew top-level access, and clarifying which flows are push vs. sheet.

Current
RootView (2 tabs)
├── Home (NavigationStack)
│   ├── push → LineupView
│   │   └── push → FestivalMapView
│   ├── sheet ↑ ArtistDetailView
│   │   └── push → FestivalMapView
│   └── sheet ↑ ConflictResolverView
└── Profile
     └── ProfileView
Recommended
RootView (5 tabs)
├── Home — festival context
│   ├── sheet ↑ ArtistDetailView
│   └── sheet ↑ ConflictResolverView
├── Lineup (NavigationStack)
│   ├── TimetableGridView / DiscoverView
│   └── sheet ↑ ArtistDetailView
├── Map — FestivalMapView
│   └── sheet ↑ StageDetailView
├── Crew (NavigationStack)
│   ├── CrewView
│   └── sheet ↑ CrewInviteView
└── Profile
     ├── sheet ↑ EditProfileView
     └── sheet ↑ NotificationsView
